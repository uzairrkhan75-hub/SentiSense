# Deploying SentiSense (Render + Vercel)

This project is a React (Vite) frontend + Flask backend + MongoDB app.

- **Frontend**: deployed to **Vercel** — zero-config, git-push deploys, global CDN, free tier.
- **Backend**: deployed to **Render** as a Docker web service, using the existing `backend/Dockerfile` as-is.
- **Database**: **MongoDB Atlas** (free M0 tier) — neither platform hosts MongoDB itself.

Code changes already made to support this (see `git log`):
- `backend/wsgi.py` — production entry point; gunicorn imports `app` from here (`run.py` is local-dev only).
- `backend/Dockerfile` — now runs `gunicorn -w 2 -b 0.0.0.0:${PORT:-5000} wsgi:app` instead of the Flask dev server, and binds to Render's dynamic `$PORT`. **Verified locally**: built the image and ran the container with gunicorn, confirmed `/api/health` returns 200 both on the default port and with `PORT` overridden.
- `backend/app/config.py` — refuses to start in production without real `SECRET_KEY`/`JWT_SECRET_KEY` env vars.
- `frontend/src/api/client.js` — API base URL now reads `VITE_API_BASE_URL` (falls back to `/api` for local dev).
- `render.yaml` — Render Blueprint defining the backend service, health check path, and env vars.
- `frontend/vercel.json` — SPA rewrite so React Router routes don't 404 on refresh.

---

## 1. MongoDB Atlas

1. Create a free M0 cluster at [mongodb.com/cloud/atlas](https://www.mongodb.com/cloud/atlas).
2. **Database Access** → add a database user with a strong password.
3. **Network Access** → add `0.0.0.0/0`. Render's outbound IPs aren't static on the free plan, so a fixed allowlist isn't practical — the DB user's password is the actual security boundary here.
4. Copy the connection string (`mongodb+srv://user:password@cluster.../sentisense`) — this is your `MONGO_URI`.

## 2. Backend — Render

**Option A: Blueprint (recommended)** — Render reads `render.yaml` automatically:
1. Push this repo to GitHub/GitLab.
2. In Render: New → Blueprint → connect the repo. It'll detect `render.yaml` and create the `sentisense-backend` web service.
3. `SECRET_KEY`/`JWT_SECRET_KEY` are auto-generated by Render (see `generateValue: true` in `render.yaml`). Fill in the `sync: false` vars manually in the Render dashboard:
   | Variable | Value |
   |---|---|
   | `MONGO_URI` | the Atlas connection string from step 1 |
   | `CORS_ORIGINS` | your Vercel frontend URL, e.g. `https://sentisense.vercel.app` |
   | `GOOGLE_CLIENT_ID` | your Google OAuth Client ID |

**Option B: Manual web service** — New → Web Service → connect repo → Environment: Docker → Dockerfile path `backend/Dockerfile`, Docker context `backend`. Set the same env vars as above, plus `FLASK_ENV=production`.

Either way:
- Render injects `$PORT` automatically — the Dockerfile already handles this.
- Health check path is `/api/health` (already set in `render.yaml`).
- Free tier spins down after inactivity; the first request after idle will be slow (cold start) — normal for the free plan.
- BERT/transformer model stays **available** here (unlike shared hosting) if you want it — just install `requirements-transformer.txt` too and don't set `SENTISENSE_DISABLE_BERT`. Skip this on the free plan though; the ~500MB+ PyTorch download will likely exceed free-tier build/memory limits. Only enable on a paid Render plan.

## 3. Frontend — Vercel

1. In Vercel: New Project → import the repo → set **Root Directory** to `frontend`.
2. Framework preset: Vite (auto-detected). Build command `npm run build`, output directory `dist` (Vercel defaults are already correct).
3. Environment variables (Vercel dashboard → Project Settings → Environment Variables):
   | Variable | Value |
   |---|---|
   | `VITE_API_BASE_URL` | your Render backend URL + `/api`, e.g. `https://sentisense-backend.onrender.com/api` |
   | `VITE_GOOGLE_CLIENT_ID` | your Google OAuth Client ID |
4. Deploy. `frontend/vercel.json` already handles SPA routing so refreshing on e.g. `/dashboard` won't 404.

## 4. Google OAuth

In Google Cloud Console, on the existing OAuth Client ID, add your Vercel URL (e.g. `https://sentisense.vercel.app`) to **Authorized JavaScript origins** (alongside the localhost ones already there for dev).

## 5. Post-deploy checklist

- [ ] `https://<your-render-app>.onrender.com/api/health` returns 200
- [ ] Register/login works end-to-end from the live Vercel frontend
- [ ] Google sign-in works from the live domain (not just localhost)
- [ ] Single-text analyze, bulk upload, and Compare all return results
- [ ] Browser devtools console has no CORS or 404 errors
- [ ] `backend/.env` / real secrets are never committed to git (already gitignored — see `git log` "Untrack node_modules..." commit)

## Notes

- Local `docker-compose.yml` still works unchanged for local Docker dev — it doesn't set `PORT`, so the Dockerfile's `${PORT:-5000}` default kicks in.
- If you outgrow Render/Vercel's free tiers, a VPS with `docker-compose.yml` is the natural next step — no code changes needed, same Dockerfiles.
